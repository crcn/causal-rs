# RFC: Macro-Based Effect and Reducer API (v0.11)

**Status:** Draft
**Version:** 0.11.0
**Date:** 2026-02-06

## Summary

Introduce `#[effect]` and `#[reducer]` procedural macros to provide a more idiomatic, function-based API alongside the existing closure-based builder pattern. This improves ergonomics, testability, and IDE support while maintaining full backwards compatibility.

## Motivation

The current closure-based API works well but has ergonomic limitations:

**Current API:**
```rust
effect::on::<OrderEvent>()
    .extract(|e| match e {
        OrderEvent::Placed { order_id, .. } => Some(*order_id),
        _ => None,
    })
    .retry(3)
    .timeout(Duration::from_secs(30))
    .then(|order_id, ctx| async move {
        ctx.deps().ship(order_id).await?;
        Ok(OrderEvent::Shipped { order_id })
    })
```

**Pain points:**
1. **Verbose** - Pattern matching and chaining creates visual noise
2. **Anonymous closures** - Poor stack traces, harder to debug
3. **Type inference** - Sometimes requires turbofish annotations
4. **Testing** - Can't call effect logic directly, must go through runtime
5. **IDE support** - Limited autocomplete/refactoring for closures

## Proposed Design

### `#[effect]` Macro

Transform async functions into effect builders:

```rust
#[effect(
    on = PaymentRequested,
    retry = 3,
    timeout_secs = 30,
    priority = 1,
    group = "payments"
)]
async fn charge_payment(
    event: PaymentRequested,
    ctx: EffectContext<OrderState, Deps>
) -> Result<PaymentCharged> {
    ctx.deps().stripe.charge(&event).await?;
    Ok(PaymentCharged { order_id: event.order_id })
}
```

### `#[reducer]` Macro

Transform pure functions into reducer builders:

```rust
#[reducer(on = OrderPlaced)]
fn order_placed(state: OrderState, event: OrderPlaced) -> OrderState {
    OrderState {
        total_orders: state.total_orders + 1,
        total_revenue: state.total_revenue + event.total,
        last_order_id: Some(event.order_id),
    }
}
```

### Module Registration

Group related effects/reducers:

```rust
#[effects]
mod order_effects {
    #[effect(on = OrderPlaced)]
    async fn ship_order(event: OrderPlaced, ctx: Context) -> Result<OrderShipped> {
        // ...
    }

    #[effect(on = OrderShipped)]
    async fn send_confirmation(event: OrderShipped, ctx: Context) -> Result<ConfirmationSent> {
        // ...
    }
}

// Generated by macro:
// impl order_effects {
//     pub fn effects<S, D>() -> Vec<Effect<S, D>> { ... }
// }

let engine = Engine::new(deps, store)
    .with_effects(order_effects::effects())
    .with_reducers(order_reducers::reducers());
```

## Detailed Specification

### Effect Attributes

```rust
#[effect(
    on = EventType | [Variant1, Variant2],  // Required: event type(s)
    extract(field1, field2),                 // Optional: field extraction
    join,                                    // Optional: batch accumulation
    id = "custom_id",                       // Optional: effect identifier
    retry = N,                              // Optional: retry count (default 1)
    timeout_secs = N,                       // Optional: timeout in seconds
    timeout_ms = N,                         // Optional: timeout in milliseconds
    delay_secs = N,                         // Optional: delay execution
    delay_ms = N,                           // Optional: delay in milliseconds
    priority = N,                           // Optional: priority (lower = higher)
    group = "name",                         // Optional: logical grouping
)]
async fn effect_name(
    /* event | Vec<event> | extracted fields */,
    ctx: EffectContext<State, Deps>
) -> Result<OutputEvent | Emit<OutputEvent>>
```

**Validation rules:**
1. Must be async function
2. Must return `Result<T>` where `T: Into<Emit<E>>`
3. If `join` is set, first parameter must be `Vec<T>` where `on = T`
4. Cannot specify both `timeout_secs` and `timeout_ms`
5. Cannot specify both `delay_secs` and `delay_ms`
6. If `extract()` is used, function parameters must match extracted field types

### Reducer Attributes

```rust
#[reducer(
    on = EventType | [Variant1, Variant2],  // Optional: specific event(s)
    extract(field1, field2),                 // Optional: field extraction
)]
fn reducer_name(
    state: State,
    event: Event | /* extracted fields */
) -> State
```

**Validation rules:**
1. Must be synchronous function (no async)
2. Must return same type as state parameter
3. Must be pure (no `ctx` parameter allowed)
4. If no `on` specified, expects pattern matching on event enum

### Field Extraction

Split trigger and extraction for cleaner syntax and better error messages:

```rust
// Multi-variant with extraction
#[effect(
    on = [CrawlEvent::Ingested, CrawlEvent::Regenerated],
    extract(website_id, job_id)
)]
async fn enqueue_extract(
    website_id: Uuid,
    job_id: Uuid,
    ctx: EffectContext<State, Deps>
) -> Result<ExtractEnqueued> {
    ctx.deps().enqueue(website_id, job_id).await?;
    Ok(ExtractEnqueued { website_id })
}
```

The macro generates code that:
1. Matches on specified variants
2. Extracts named fields
3. Passes extracted values to function
4. Rustc validates field presence and types at compile time

### Join Pattern

```rust
#[effect(on = RowValidated, join)]
async fn bulk_insert(
    batch: Vec<RowValidated>,
    ctx: EffectContext<State, Deps>
) -> Result<BatchInserted> {
    ctx.deps().bulk_insert(&batch).await?;
    Ok(BatchInserted { count: batch.len() })
}
```

**Join contract:**
- First parameter must be `Vec<T>` where `on = T`
- Accumulates all events with same `batch_id`
- Handler fires when all events in batch arrive

### Pattern Matching Reducers

For reducers handling multiple event variants:

```rust
#[reducer]
fn order_lifecycle(state: OrderState, event: OrderEvent) -> OrderState {
    match event {
        OrderEvent::Placed { total, order_id } => OrderState {
            total_orders: state.total_orders + 1,
            total_revenue: state.total_revenue + total,
            last_order_id: Some(order_id),
        },
        OrderEvent::Shipped { .. } => state,
        OrderEvent::Delivered { .. } => OrderState {
            active_orders: state.active_orders - 1,
            ..state
        },
        OrderEvent::Cancelled { .. } => OrderState {
            active_orders: state.active_orders - 1,
            cancelled_orders: state.cancelled_orders + 1,
            ..state
        },
    }
}
```

When no `on` attribute is specified, the macro expects a match statement on the event parameter.

## Code Generation

### Effect Generation

```rust
// User writes:
#[effect(on = OrderPlaced, retry = 3)]
async fn ship_order(event: OrderPlaced, ctx: EffectContext<State, Deps>) -> Result<OrderShipped> {
    ctx.deps().ship(event.order_id).await?;
    Ok(OrderShipped { order_id: event.order_id })
}

// Macro generates:
#[doc(hidden)]
pub fn __seesaw_effect_ship_order<S, D>() -> Effect<S, D>
where
    S: Clone + Send + Sync + 'static,
    D: Clone + Send + Sync + 'static,
{
    effect::on::<OrderPlaced>()
        .retry(3)
        .then(|event, ctx| ship_order(event, ctx))
}

// User function unchanged - can be called directly for testing
async fn ship_order(event: OrderPlaced, ctx: EffectContext<State, Deps>) -> Result<OrderShipped> {
    ctx.deps().ship(event.order_id).await?;
    Ok(OrderShipped { order_id: event.order_id })
}
```

### Module Generation

```rust
// User writes:
#[effects]
mod order_effects {
    #[effect(on = OrderPlaced)]
    async fn ship_order(event: OrderPlaced, ctx: Context) -> Result<OrderShipped> { ... }

    #[effect(on = OrderShipped)]
    async fn notify(event: OrderShipped, ctx: Context) -> Result<NotificationSent> { ... }
}

// Macro generates:
mod order_effects {
    // User functions stay as-is
    async fn ship_order(event: OrderPlaced, ctx: Context) -> Result<OrderShipped> { ... }
    async fn notify(event: OrderShipped, ctx: Context) -> Result<NotificationSent> { ... }

    // Generated hidden wrappers
    #[doc(hidden)]
    fn __seesaw_effect_ship_order<S, D>() -> Effect<S, D> { ... }
    #[doc(hidden)]
    fn __seesaw_effect_notify<S, D>() -> Effect<S, D> { ... }

    // Generated registration function
    pub fn effects<S, D>() -> Vec<Effect<S, D>>
    where
        S: Clone + Send + Sync + 'static,
        D: Clone + Send + Sync + 'static,
    {
        vec![
            __seesaw_effect_ship_order(),
            __seesaw_effect_notify(),
        ]
    }
}
```

## Examples

### Complete Application

```rust
use seesaw_core::{effect, effects, reducer, reducers, Emit, Engine, EffectContext};
use seesaw_memory::MemoryStore;

// State
#[derive(Clone, Default)]
struct OrderState {
    total_orders: u32,
    total_revenue: f64,
    active_orders: u32,
}

// Events
#[derive(Clone)]
enum OrderEvent {
    Placed { order_id: Uuid, total: f64 },
    Shipped { order_id: Uuid },
    Delivered { order_id: Uuid },
}

// Reducers module
#[reducers]
mod state {
    use super::*;

    #[reducer(on = OrderPlaced)]
    fn order_placed(state: OrderState, event: OrderPlaced) -> OrderState {
        OrderState {
            total_orders: state.total_orders + 1,
            total_revenue: state.total_revenue + event.total,
            active_orders: state.active_orders + 1,
        }
    }

    #[reducer(on = OrderDelivered)]
    fn order_delivered(state: OrderState, _event: OrderDelivered) -> OrderState {
        OrderState {
            active_orders: state.active_orders - 1,
            ..state
        }
    }
}

// Effects module
#[effects]
mod effects {
    use super::*;

    #[effect(on = OrderPlaced)]
    async fn ship_order(
        event: OrderPlaced,
        ctx: EffectContext<OrderState, Deps>
    ) -> Result<OrderShipped> {
        ctx.deps().shipping.ship(event.order_id).await?;
        Ok(OrderShipped { order_id: event.order_id })
    }

    #[effect(on = OrderShipped)]
    async fn send_notification(
        event: OrderShipped,
        ctx: EffectContext<OrderState, Deps>
    ) -> Result<OrderDelivered> {
        ctx.deps().email.send_shipped_notification(event.order_id).await?;
        Ok(OrderDelivered { order_id: event.order_id })
    }

    #[effect(on = OrderPlaced, group = "analytics")]
    async fn track_order(
        event: OrderPlaced,
        ctx: EffectContext<OrderState, Deps>
    ) -> Result<Emit<AnalyticsEvent>> {
        ctx.deps().analytics.track("order_placed", &event);
        Ok(Emit::None)
    }
}

#[tokio::main]
async fn main() -> Result<()> {
    let store = MemoryStore::new();
    let deps = Deps::new();

    let engine = Engine::new(deps, store)
        .with_reducers(state::reducers())
        .with_effects(effects::effects());

    let order_id = Uuid::new_v4();
    engine.process(OrderEvent::Placed { order_id, total: 99.99 }).await?;

    Ok(())
}
```

### Batch Processing with Join

```rust
#[effects]
mod import_effects {
    #[effect(on = FileUploaded)]
    async fn parse_file(
        event: FileUploaded,
        ctx: EffectContext<State, Deps>
    ) -> Result<Emit<ImportEvent>> {
        let rows = ctx.deps().parse_csv(&event.path).await?;

        let events = rows.into_iter()
            .map(|data| ImportEvent::RowParsed {
                row_id: Uuid::new_v4(),
                data
            })
            .collect();

        Ok(Emit::Batch(events))
    }

    #[effect(on = RowParsed)]
    async fn validate_row(
        event: RowParsed,
        ctx: EffectContext<State, Deps>
    ) -> Result<RowValidated> {
        ctx.deps().validate(&event.data).await?;
        Ok(RowValidated {
            row_id: event.row_id,
            data: event.data
        })
    }

    #[effect(on = RowValidated, join)]
    async fn bulk_insert(
        batch: Vec<RowValidated>,
        ctx: EffectContext<State, Deps>
    ) -> Result<BatchInserted> {
        let rows: Vec<_> = batch.into_iter().map(|r| r.data).collect();
        ctx.deps().bulk_insert(&rows).await?;
        Ok(BatchInserted { count: rows.len() })
    }
}
```

### Multi-Variant Extraction


## Testing

### Direct Function Testing

```rust

async fn ship_order(
    event: OrderPlaced,
    ctx: EffectContext<State, Deps>
) -> Result<OrderShipped> {
    ctx.deps().ship(event.order_id).await?;

// Test the function directly
#[tokio::test]
async fn test_ship_order() {
    let event = OrderPlaced {
        order_id: Uuid::new_v4(),
        total: 99.99
    };
    let ctx = mock_context();

    let result = ship_order(event, ctx).await;

    assert!(result.is_ok());
    assert_eq!(result.unwrap().order_id, event.order_id);
}
```

### Integration Testing

```rust
#[tokio::test]
async fn test_order_flow() {
```

## Migration Path

### Phase 1: Introduction (v0.11.0)

1. Add proc macros as optional feature
2. Update documentation with macro examples
3. Keep closure API as primary in examples
4. Mark closure API as "stable, not deprecated"

### Phase 2: Adoption (v0.12.0)

1. Use macro API in all new examples
2. Add migration guide
3. Show both APIs side-by-side in docs
4. Community feedback and iteration

### Phase 3: Maturity (v0.13.0+)

1. Macro API becomes primary in documentation
2. Closure API documented as "advanced/when needed"
3. Consider deprecation path for closure API (far future, maybe never)

## Backwards Compatibility

**Fully backwards compatible:**
- Closure API remains unchanged
- No breaking changes to existing code
- Macro API is additive only
- Both APIs can be mixed in same codebase

```rust
// Mix and match as needed
let engine = Engine::new(deps, store)
    .with_effects(order_effects::effects())  // Macro
    .with_effect(  // Closure (when you need complex composition)
        effect::on::<ComplexEvent>()
            .filter(|e| e.is_valid())
            .transition(|prev, next| prev.status != next.status)
            .retry(5)
            .timeout_secs(60)
            .then(|event, ctx| async move { ... })
    );
```

## Open Questions

1. **Enum method syntax** - Should we support `#[reducer]` on enum methods?
   ```rust
   impl OrderEvent {
       #[reducer]
       fn reduce(&self, state: OrderState) -> OrderState { ... }
   }
   ```

2. **Filter predicates** - How to express filters in macro form?
   ```rust
   #[effect(on = OrderPlaced, filter = large_orders)]
   fn large_orders(event: &OrderPlaced) -> bool { event.total > 100.0 }
   ```

3. **Transition guards** - How to express state transitions?
   ```rust
   #[effect(on = StatusChanged, transition = status_changed)]
   fn status_changed(prev: &State, next: &State) -> bool {
       prev.status != next.status
   }
   ```

4. **Auto-registration** - Should modules auto-register without explicit call?
   ```rust
   #[effects(register)]  // Auto-adds to some global registry?
   mod order_effects { ... }
   ```

## Alternatives Considered

### 1. Keep closure API only
**Rejected** - Community feedback shows strong preference for function-based APIs in modern Rust.

### 2. Trait-based approach
```rust
impl Effect<OrderState, Deps> for ShipOrder {
    type Event = OrderPlaced;
    type Output = OrderShipped;

    async fn handle(&self, event: Self::Event, ctx: Context) -> Result<Self::Output>
}
```
**Rejected** - More boilerplate than macros, harder to compose.

### 3. DSL approach
```rust
effect! {
    on OrderPlaced with retry = 3, timeout = 30 {
        |event, ctx| async move { ... }
    }
}
```
**Rejected** - Not idiomatic Rust, poor IDE support, harder to learn.

## Future Enhancements

### v0.12+: Effect Composition

```rust
#[effect(on = OrderPlaced, compose = [validate_order, charge_payment])]
async fn process_order(event: OrderPlaced, ctx: Context) -> Result<OrderProcessed>
```

### v0.12+: Conditional Effects

```rust
#[effect(on = OrderPlaced, when = |e| e.total > 100.0)]
async fn process_large_order(event: OrderPlaced, ctx: Context) -> Result<OrderProcessed>
```

### v0.13+: Saga Pattern

```rust
#[saga]
async fn order_saga(order: Order, ctx: Context) -> Result<OrderComplete> {
    let payment = charge_payment(order, ctx).await?;
    let shipment = ship_order(order, ctx).await
        .compensate(refund_payment(payment, ctx))?;
    Ok(OrderComplete { order, payment, shipment })
}
```

## Conclusion

The macro-based API provides:
- ✅ Better ergonomics and readability
- ✅ Improved testability
- ✅ Better IDE support
- ✅ Cleaner stack traces
- ✅ Idiomatic Rust patterns
- ✅ Full backwards compatibility

This positions Seesaw as a modern, developer-friendly event-driven runtime while maintaining the power and flexibility of the existing closure-based API.
